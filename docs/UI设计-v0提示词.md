# RAG 智能知识平台 — v0 / 外部 AI UI 设计提示词

> **用途**：框架与达交逻辑敲定后，用 [v0.app](https://v0.app/) 或其他 AI 做 UI 视觉规划；产出交给工程侧融合进现有 Vue3 工程。  
> **技术栈（最终实现）**：Vue 3 + Vite + Element Plus + Pinia + Axios（**不是** React / Next / shadcn 生产代码）  
> **策略**：先短提示词（Login + Layout）定骨架 → 再长提示词做全站规范与分批扩展  
> **日期**：2026-07-18

---

## 0. 使用说明（给设计方 / 产品方）

### 0.1 推荐顺序

| 步骤 | 做什么 | 用哪份提示词 |
|:----:|--------|--------------|
| 1 | 只做 Login + 全局 Layout，定品牌与导航骨架 | **短提示词**（§1） |
| 2 | 确认视觉方向（配色、密度、品牌感） | 人工评审截图/预览 |
| 3 | 按页分批扩展（HitTest / Chat / Branding / 管理页…） | **长提示词**（§2）作规范附件 |
| 4 | 把「截图 + Tokens + 组件映射 + 对接 Checklist」交给工程 AI | 见 §3 |

### 0.2 交付物要求（设计 AI 必须交回）

1. Design Tokens（色/字/间距/圆角）  
2. 线框或高保真说明（可附预览截图）  
3. Element Plus 组件映射表  
4. 对接 Checklist（哪些纯样式、哪些绝对不能改字段）  
5. 若生成了 React/Tailwind 代码：注明「仅预览，需改写为 Vue SFC」

### 0.3 工程侧融合原则

- 只改 UI 层（`views` / `styles` / 主题变量），**尽量不改 API 契约**  
- 保留 Pinia、Axios 拦截、文档状态、SSE、权限菜单等既有逻辑  
- v0 代码**禁止整仓替换**

## 1. 短提示词（仅 Login + Layout）

> 复制下面代码块全文，粘贴到 v0 或其他设计 AI。

```text
请只设计 2 个界面的高保真 UI 方案（不要做全站，不要改业务契约）。

【技术边界】
- 最终实现：Vue 3 + Element Plus + Pinia（不是 React 生产代码）
- 你可生成预览，但必须附 Element Plus 映射说明
- 禁止改路由/API/字段名；禁止新造接口

【页面 1：/login】
- 品牌优先：系统名、Logo、登录标语是主角（用占位变量，勿写死）
  brand_name / brand_logo_url / brand_login_title / brand_theme_color
- 仅保留：用户名、密码、登录按钮；可保留员工/管理员切换（若有）
- 首屏不要统计卡片、徽章、营销贴纸
- 桌面优先；背景可有克制氛围，避免紫白渐变/厚重 glow/emoji

【页面 2：全局 Layout】
- 左导航 + 顶栏 + 主内容区 + 页脚
- 侧栏：Logo + brand_name；页脚：brand_footer_text
- 菜单（按现有信息架构，勿新增路由）：
  系统概览 / 角色 / 用户 / 用户组 / 大模型 / 品牌配置 /
  知识库 / 文档 / 命中测试 / 智能对话
- 标注 admin 与普通用户菜单可见差异即可（不要做复杂权限矩阵）
  - admin：管理后台菜单全开 + RAG 核心页
  - user（普通用户）：知识库 / 文档 / 命中测试 / 智能对话（数据受用户组隔离）；不见用户/角色/用户组/品牌等管理页
- 顶栏：标题区 + 当前角色 + 退出

【视觉约束】
- 先给 Design Tokens（色/字/间距/圆角）
- 主题色可替换（默认可参考 #409EFF）
- 避免 AI 俗套审美；动效最多 1–2 个

【交付格式】
1) Tokens 表
2) Login 线框 + 高保真说明 + Element Plus 映射
3) Layout 线框 + 高保真说明 + 菜单结构
4) 给工程对接的 Checklist（哪些纯样式可换、哪些绝对不能改）
5) 若输出了 React/Tailwind 预览，注明“仅预览，需改写成 Vue SFC”
```

## 2. 长提示词（全站规范 + 分批扩展）

> 在短提示词方向确认后使用。可整段投喂，或拆成「规范附录 + 单页任务」。  
> 建议每次只让 AI 深做 1～2 个关键页，长文作为约束，避免一次摊大饼。

```text
你是资深产品设计师 + 前端视觉顾问。请为本项目做 UI 设计规划与高保真视觉方案。

【非常重要：工作边界】
1. 本仓库最终实现技术栈是：Vue 3 + Vite + Element Plus + Pinia + Axios（不是 React / Next / shadcn）。
2. 你的产出是“设计规格 + 视觉稿 + Element Plus 组件映射”，不是可直接替换仓库的生产代码。
3. 允许用任意技术生成预览界面（包括 React），但必须同时给出“如何映射到 Element Plus / Vue”的说明。
4. 禁止改变现有业务契约：路由 path、API path、请求/响应字段名、权限模型、文档状态枚举。
5. 禁止新造后端接口；所有交互只能绑定现有能力。
6. 不要做花哨的多卡片仪表盘堆砌；管理后台保持清晰、可操作、桌面端优先。
7. 避免常见 AI 审美套路：紫白渐变、奶油底+衬线+陶土色、报纸排版、过度 glow、到处 rounded-full pill、多层厚阴影、emoji 装饰。

【产品定位】
RAG 智能知识平台（企业知识库管理 + 检索命中测试 + 智能对话 + 用户组权限隔离 + 白标 OEM）。
角色模型（两级，勿发明第三角色）：
- admin（管理员）：系统配置与管理后台
- user（普通用户）：仅访问用户组授权范围内的知识库 / 文档 / 命中测试 / 对话

【必须覆盖的页面（与现有路由一一对应，不要新增路由）】
- /login — 登录页
- /dashboard — 系统概览
- /roles — 角色管理
- /users — 用户管理（需支持分配用户组多选）
- /user-groups — 用户组管理（成员管理 + 知识库授权穿梭框）
- /models — 大模型配置
- /branding — 品牌/白标配置（名称、Logo、favicon、主题色、登录标语、页脚）
- /knowledge-bases — 知识库管理
- /documents — 文档管理（现状是扁平路由；设计时按“先选知识库再管文档”）
- /hit-test — 命中率测试
- /chat — 智能对话（SSE 流式）
- 全局 Layout：左侧导航 + 顶栏 + 内容区 + 页脚

【白标（必须可替换，不要写死品牌）】
设计中所有品牌位用占位变量标注：
- brand_name
- brand_logo_url
- brand_favicon_url
- brand_theme_color（默认可参考 #409EFF，但要可替换）
- brand_login_title
- brand_footer_text
生效范围：浏览器标题/favicon、侧栏 Logo+名称、登录标语、主题色、页脚、对话窗口标题（系统名 + 智能助手）。

【不可改的业务与字段契约】
1. 检索/对话输入字段名必须是 query（不要用 question/message/content）。
2. 文档状态仅四种：pending / processing / completed / failed。
   - 非 completed：下拉禁用 + 状态标签；命中测试运行前要有明确等待/失败提示。
3. 检索方式仅三种：vector / keyword / hybrid。
4. 统一错误提示需考虑这些业务码：0/400/401/403/404/500/5001/5002/4001/4002/4003。
5. 角色简化为两级：admin（管理员）与 user（普通用户）；不要设计复杂细粒度权限矩阵 UI；不要“编辑员”。
6. Chat 是 SSE 流式输出，要有进行中气泡、停止、来源引用区。
7. Dashboard：知识库总数、文档总数、用户总数；调用次数 call_count 仅留空壳占位（可显示 “—” 或“二期”）。
8. 普通用户菜单可见：知识库、文档、命中测试、智能对话（数据隔离由后端保证）；管理页仅 admin。

【视觉与布局要求】
1. 管理后台：左导航 + 右内容，信息密度适中，表格/表单/弹窗为主。
2. 登录页：品牌优先（名称/Logo/标语是主角），不要堆统计卡片。
3. 对话页：会话列表 + 消息流 + 输入区，清晰展示引用片段与相关度。
4. 命中测试页：知识库→文档级联、检索方式、query、结果列表（内容/分数/来源）。
5. 响应式：桌面优先；宽度 <1280 不崩即可，不必做移动端完整重构。
6. 请先定义一套 Design Tokens（颜色、字号、间距、圆角、边框、阴影），再套到各页，保证全站一致。
7. 动效克制：允许 2–3 个有意义动效（如页面进入、流式文字、主题色切换），不要炫技。

【你必须交付的内容（按这个结构输出）】
A. Design Tokens 表（名称、值、用途）
B. 全局 Layout 规格（导航信息架构、角色可见菜单差异）
C. 逐页说明（每个页面包含）：
   - 布局线框文字版（区块从上到下）
   - 关键组件列表
   - Element Plus 组件映射（例如：表格→el-table，弹窗→el-dialog，穿梭框→el-transfer，色板→el-color-picker，上传→el-upload）
   - 状态设计：loading / empty / error / 无权限 / 文档未就绪
   - 与现有页面文件的对应建议名（如 Login.vue、Layout.vue、HitTest.vue…）
D. 高保真预览：优先做这 6 个关键页
   1) Login  2) Layout+Dashboard  3) UserGroupManage  4) BrandingConfig  5) HitTest  6) ChatDialog
   其余页面给简化版一致性方案即可。
E. “对接清单（给工程 AI）”：列出哪些是纯样式可换、哪些涉及交互文案、哪些绝对不能改字段。
F. 明确标注“非最终代码”：若你生成了 React/Tailwind 预览代码，请附加一节 Mapping Notes，说明如何改写成 Vue SFC + Element Plus，不要假设会被直接合并进仓库。

【明确不要做的事】
- 不要改 API 路径或新增 REST 资源
- 不要把文档路由设计成必须依赖未知后端能力
- 不要引入暗黑模式作为默认（除非做成可选且不影响白标主题色）
- 不要用无意义的统计卡片、营销徽章、浮动 sticker 堆砌首屏
- 不要输出“整仓替换方案”
- 不要发明第三角色（如编辑员）

【开始方式】
先给出 Design Tokens + 信息架构，再依次产出 6 个关键页高保真方案与 Element Plus 映射；最后给一份给我方工程对接用的 Checklist。
若本次任务被限定为单页，则只深做指定页，但仍须遵守本文全部契约与 Tokens。
```

## 3. 交回工程侧时的打包清单

把 v0 / 其他 AI 的产出按页打包时，建议包含：

| 项 | 说明 |
|----|------|
| 对应路由 / 现有 `*.vue` 文件名 | 例如 `/login` → `Login.vue` |
| 截图或预览链接 | 桌面宽度为主 |
| Tokens 增量 | 相对默认主题改了哪些变量 |
| Element Plus 映射 | 设计组件 → `el-*` |
| 白标占位是否齐全 | 六项 config_key |
| 契约自检 | `query`、文档四态、两级角色、未新增路由 |
| 预览代码（可选） | 标注仅供参考，勿直接合入 |
