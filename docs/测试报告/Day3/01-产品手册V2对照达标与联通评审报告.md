## 4. 前后端联通评审

### 4.1 已打通（主链路）

| # | 链路 | 证据要点 |
|---|------|----------|
| 1 | 登录 → JWT | `POST /api/auth/login` ↔ `frontend/src/api/auth.js`（兼容 `access_token`） |
| 2 | 知识库 CRUD | `KbManage` / `api/kb` ↔ `api/kb.py` |
| 3 | 文档上传/列表/删除 | `DocManage` / `api/doc` ↔ `api/docs.py` |
| 4 | 命中测试 | `POST /api/rag/test_retrieve` ↔ HitTest |
| 5 | 智能对话 SSE | `POST /api/chat/stream` ↔ `useSSE` / chat store |
| 6 | 用户组 / 品牌 / 仪表盘 | 管理页有对应 API 封装 |

### 4.2 未完全打通 / 联调风险

| # | 问题 | 影响 | 级别 |
|---|------|------|:----:|
| 1 | 登录成功不返回 `user`，前端兜底 `role_name: '管理员'` | 菜单“碰巧能开”，非真实角色 | P0 |
| 2 | 后端角色名 `admin`/`user`，前端菜单认中文角色 | 普通用户菜单体系不可靠 | P0 |
| 3 | 缺 `GET /api/auth/me` | 无法用 Token 拉真实用户/角色 | P0 |
| 4 | 管理接口多仅 `get_current_user`，未强制 admin | 任意登录用户可调写接口 | P0 |
| 5 | `require_kb_access` 抛 HTTP 403，非统一体 `code:403` | 前端业务码拦截可能漏提示 | P1 |
| 6 | Chat 会话 API 默认 `VITE_CHAT_SESSION_API` 关闭 | 多轮历史未真正走后端 | P1 |
| 7 | Logo URL `/uploads/branding/...` 未挂 StaticFiles | 白标图片链不通 | P1 |
| 8 | `userGroups`/`branding` 失败静默 Mock | 联调易误判“已打通” | P1 |
| 9 | 文档路由手册嵌套 vs 实际扁平 `/documents` | 契约不一致（功能可绕过） | P2 |

**联通结论：主业务链路可通；身份角色、菜单权限、白标资源、会话持久化未全通。**

| code | 手册含义 | 落地 | 备注 |
|:----:|----------|:----:|------|

| 404 | 不存在 | ✅ | — |
| 500 | 内部异常 | ⚠️ | 部分显式；无全局统一包装 |

## 7. 矛盾 / 冲突明细

### 7.1 产品手册自身不一致

1. **表数量**：§4.2 写「共 12 张表」，列出实体约 **11** 张；验收标准写 11 张。
2. **文档 API 路径**：§5.4 表头与表内路径混用（嵌套 KB 路径 vs `/api/documents*`）。
3. **4003 含义**：手册 §5.3 = 白标配置缺失；部分契约/测试语境又作「字段必须为 query」。
4. **conversations 字段**：手册 `query`/`answer` 列 vs ORM `role`/`content` 消息行。
5. **用户状态**：手册 `active`/`disabled` vs 实现中文状态。

### 7.2 手册 ↔ 实现冲突（摘要）

| # | 手册要求 | 现状 |
|---|----------|------|
| 1 | 输入统一 `query` | Chat 用 `question` |
| 2 | Doc 路由嵌套 `:id/documents` | `/documents` |
| 3 | 支持 `.pdf` | 仅 md/txt |
| 4 | DocManage 3s 轮询 | 无 |
| 5 | HitTest ElMessageBox 等待提示 | 弱提示 |
| 6 | Dashboard `call_count` 空壳 | 无 |
| 7 | `GET /api/auth/me` | 无 |
| 8 | 中文角色/菜单体系 | 种子 `admin`/`user` |
| 9 | 统一异常体 11 码 | 401/403 形态分裂 |
| 10 | 白标覆盖登录/侧栏/页脚 | 硬编码 |
| 11 | 自定义 Prometheus 业务指标 | 未真正采集 |
| 12 | 用户管理分配用户组 | 前端无 |

### 7.3 前后端互相对不上

1. 角色名：`admin` vs `管理员`
2. 登录响应：`access_token` only vs 期望 `token` + `user`
3. Dashboard 字段：`group_count` vs 手册 `call_count`
4. KB 删除与文档删除的向量清理不对称
5. branding/userGroups 静默 Mock 掩盖失败

### 7.4 安全 / 权限（交付风险最高）

- 手册：admin 才可用户/角色/用户组/品牌/KB 写操作  
- 实现：多数接口只要登录即可；前端菜单用中文角色名“挡一下”，**可被 API 绕过**  
- 非 admin 的 KB **读隔离有、写隔离不完整**
